# Speech-to-Speech Workflow

This document describes how data flows through the application—from the
microphone input in the browser to the spoken response generated by the
backend. Use it as a reference when debugging or extending the pipeline.

## High-Level Pipeline

```
Microphone (browser)
   ↓ MediaRecorder (WebM/Opus chunks)
WebSocket (ws://localhost:8000/ws)
   ↓ ffmpeg conversion (WebM → 16 kHz PCM)
Vosk ASR (transcription)
   ↓
Ollama LLM (agent reasoning)
   ↓
TTS (Coqui/pyttsx3) → audio output
   ↓
Browser audio player
```

## Detailed Steps

### 1. Browser capture and upload
- The React frontend requests microphone access via `navigator.mediaDevices.getUserMedia`.
- `MediaRecorder` captures audio as `audio/webm` chunks (~1 s) and pushes each chunk to the backend WebSocket encoded as Base64 JSON payloads (`{ type: "audio_chunk", audio: "<b64>" }`).
- The UI also sends typed messages using `{ type: "text_input" }` packets, and listens for server events (`transcription`, `llm_response`, `audio_output`, `error`) to keep the conversation panel in sync.

### 2. WebSocket ingestion
- FastAPI keeps a persistent `/ws` endpoint. On the first connection it loads the Vosk model, Ollama client, and TTS engine.
- Each message is routed by `type`. When an `audio_chunk` arrives, the backend validates the payload and hands it to the audio conversion helper.

### 3. Audio conversion (ffmpeg)
- Browser chunks are WebM/Opus, which Vosk cannot read directly. The helper (`utils.audio_utils.webm_base64_to_pcm`) does the following:
  1. Base64-decodes the payload.
  2. Pipes the bytes through `ffmpeg -i pipe:0 -f s16le -ac 1 -ar 16000 pipe:1`.
  3. Returns raw PCM bytes (16-bit, mono, 16 kHz).
- **Requirement:** ffmpeg must be installed and the `bin` directory must be on PATH.

### 4. Speech recognition (Vosk)
- PCM data is fed to `VoskASR.transcribe_audio`, which emits the recognized text.
- The backend immediately echoes this text to the frontend via `{ type: "transcription" }`, so the user sees what was understood.
- If the chunk contains no speech, an error packet is sent instead.

### 5. Language model response (Ollama)
- The same transcript is passed into `handle_user_text`, which calls `OllamaLLM.generate_response` using the configured model (default `mistral`).
- The textual reply is sent to the frontend as `{ type: "llm_response" }` and appended to the conversation history.

### 6. Text-to-speech playback
- The response text is synthesized through the TTS engine (Coqui TTS when available, otherwise pyttsx3).
- The resulting WAV bytes are Base64-encoded and delivered over the WebSocket via `{ type: "audio_output" }`.
- The frontend decodes this payload and plays it through an `<audio>` element, keeping latency low by avoiding extra HTTP round trips.

## Supporting Components

- **Conversation history:** Stored on the frontend and mirrored in the LLM module. Sending `{ type: "clear_history" }` resets both sides.
- **Configuration:** The backend loads `backend/config/agent_config.json` at startup and whenever `/config` is updated, letting you change the agent’s role or tone without restarting the server.
- **Fallback behavior:** If Coqui TTS is not installed, pyttsx3 provides a local Windows voice so the “speech-to-speech” loop still completes.

## Operational Notes

- Always start the backend (`python backend/main.py`) before the frontend (`npm start`) so the WebSocket endpoint is available.
- Ensure ffmpeg, Vosk models, and an Ollama model (e.g., `ollama pull mistral`) are installed before testing.
- When extending the system (e.g., streaming ASR, different languages, or new frontends) keep the conversion + transcription interface stable so only one component needs to change at a time.
